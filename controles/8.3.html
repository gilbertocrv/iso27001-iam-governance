CONTROLE · 8.3

Information Access Restriction

Define o que o usuário autenticado pode enxergar — restrição por classificação da informação, need-to-know, data masking e segregação lógica de recursos.

8.3 · Information Access Restriction 5.15 · Access Control Camada: Privilégio
Conceito Central
O que o controle exige

O que está em jogo antes de qualquer avaliação de maturidade.

O controle 8.3 exige que o acesso à informação seja restrito ao estritamente necessário para a função do usuário — mesmo dentro de um ambiente já autenticado. A autenticação prova quem você é. O 8.3 determina o que você pode ver.

Isso significa que a restrição precisa ser aplicada na camada de aplicação e dados, não apenas no diretório. Estar na rede corporativa, ter uma conta ativa e ter passado pelo MFA não são justificativas suficientes para ter acesso a qualquer recurso. Cada dado acessado precisa ter necessidade de negócio documentada para o perfil que o acessa.

8.5 vs 8.3
A diferença entre autenticar e restringir

Os dois controles são complementares — cada um protege uma camada diferente.

CONTROLE 8.5
Secure Authentication
  • Prova que você é quem diz ser
  • MFA, sessão, SSO, FIDO2
  • Aplicado na entrada do sistema
  • Pergunta: quem é você?
CONTROLE 8.3
Information Access Restriction
  • Define o que você pode enxergar
  • Mascaramento, segregação lógica, need-to-know
  • Aplicado na camada de dados e aplicação
  • Pergunta: o que você precisa ver?
Mecanismos de Restrição
Implementações que sustentam o controle

Cada mecanismo aplica o princípio de need-to-know em uma camada diferente do ambiente.

🎭

Data Masking

Dados sensíveis exibidos parcialmente ou ofuscados para perfis sem necessidade do dado aberto. CPF, número de conta, dados de saúde — visíveis apenas para funções com justificativa de negócio documentada.

🗂️

Segregação Lógica

Acesso à rede não implica acesso ao banco de dados. Sistemas segmentados por classificação da informação — um usuário de TI não enxerga o menu de Folha de Pagamento mesmo estando na mesma rede.

📋

Controle Baseado em Função

Restrição aplicada pelo perfil RBAC — o que o cargo vê, não o que o sistema permite por padrão. Configuração feita na camada de aplicação com revisão periódica por Information Owner.

🏷️

Classificação da Informação

Acesso concedido com base na classificação do dado: público, interno, confidencial ou restrito. A classificação define quem pode ver, editar, exportar e compartilhar — com enforcement técnico, não apenas documental.

Avaliação de Maturidade
Critérios por Nível

Identifique em qual nível sua organização se encontra.

NÍVEL 0–1 · INEXISTENTE / AD-HOC
Sem restrição por classificação
  • Acesso à rede equivale a acesso a tudo
  • Dados sensíveis visíveis sem controle de perfil
  • Sem classificação formal da informação
  • Restrições aplicadas apenas pontualmente, sem padrão
NÍVEL 2 · DOCUMENTADO
Classificação existe, enforcement parcial
  • Política de classificação documentada
  • Aplicação inconsistente entre sistemas
  • Data masking em alguns sistemas, não todos
  • Sem revisão periódica por Information Owner
NÍVEL 3 · PADRONIZADO
Restrição aplicada consistentemente
  • Classificação da informação definida e aplicada
  • Controles de acesso configurados por nível de classificação
  • Data masking ativo para dados sensíveis
  • Segregação lógica entre sistemas de diferentes criticidades
  • Information Owner valida acessos periodicamente
NÍVEL 4–5 · AUTOMATIZADO / RISCO
Restrição dinâmica baseada em risco e contexto
  • Controle de acesso baseado em atributos (ABAC)
  • DLP integrado com classificação automática de dados
  • Alertas em tempo real para acesso a dados fora do perfil
  • Revisão automática de permissões por mudança de contexto
  • Métricas de exposição de dados monitoradas continuamente
Riscos Associados
O que pode dar errado

Falhas de restrição de acesso e seu impacto em auditorias, incidentes e conformidade com LGPD.

ALTO · EXPOSIÇÃO
Dado sensível visível além da necessidade
Usuário acessa dados pessoais, financeiros ou estratégicos sem necessidade de negócio documentada. Além de violar 8.3, pode configurar infração à LGPD — o acesso indevido a dado pessoal é tratamento sem base legal.
ALTO · SEGREGAÇÃO
Ausência de segregação lógica entre sistemas
Acesso à infraestrutura implica visibilidade de dados de negócio. Sysadmin com acesso ao banco de dados de RH sem necessidade funcional — combinação que viola need-to-know e é achado recorrente em auditorias.
MÉDIO · MASCARAMENTO
Dado mascarado apenas na interface, não na camada de dados
Interface exibe CPF mascarado, mas a mesma conta tem acesso direto ao banco de dados. O controle existe visualmente mas não tecnicamente — não resiste a uma auditoria de camada técnica.
MÉDIO · REVISÃO
Information Owner sem revisão periódica
Classificação definida, mas sem processo de revisão formal pelo responsável pelos dados. Acessos acumulados ao longo do tempo sem validação de necessidade atual — dado disponível além do necessário sem ninguém responsável por revisar.
