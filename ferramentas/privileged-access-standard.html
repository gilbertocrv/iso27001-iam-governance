Controle 8.2 — Privileged Access Rights
Classificação de Contas Privilegiadas

Todo acesso com capacidade de causar dano irreversível deve ser classificado e tratado com controles proporcionais ao impacto potencial.

TIER 1 · CRÍTICO
Impacto Total
Pode comprometer toda a infraestrutura
Global Admin · Domain Admin · Root · Break Glass
  • Conta dedicada — nunca usada para tarefas cotidianas
  • MFA obrigatório com FIDO2 ou certificado
  • Credencial no vault com rotação automática
  • Sessão gravada e log imutável
  • Revisão mensal com evidência
  • JIT obrigatório — sem acesso permanente ativo
TIER 2 · ALTO
Alto Impacto
Pode comprometer sistemas ou dados críticos
Cloud Owner · DBA · Firewall Admin · App Admin
  • Conta dedicada separada da conta padrão
  • MFA forte obrigatório (Authenticator + number matching)
  • Credencial no vault — sem senha pessoal guardada
  • Log de sessão retido por no mínimo 90 dias
  • Revisão trimestral com evidência
TIER 3 · ELEVADO
Privilégio Funcional
Acesso amplo dentro de um escopo definido
App Admin Local · Backup Op · Deploy Prod · Read-All
  • MFA obrigatório
  • Aprovação formal para concessão
  • Revisão semestral com evidência
  • Vault recomendado para credenciais técnicas
Princípio do menor privilégio: Toda conta privilegiada deve ser usada exclusivamente para a finalidade que justificou sua concessão. Uso de conta Tier 1 para tarefas rotineiras é uma violação explícita deste padrão — independente do cargo do titular.

Gestão de Credenciais Privilegiadas
Vault e PAM — O que implementar

Padrão para custódia de credenciais administrativas. Nenhuma senha privilegiada deve existir fora de um cofre gerenciado.

PROIBIDO · PRÁTICAS INSEGURAS
O que nunca deve ocorrer
  • Senha de admin salva no navegador pessoal
  • Credencial privilegiada em planilha ou documento
  • Senha de root compartilhada por e-mail ou chat
  • Conta global admin usada para acesso cotidiano
  • Credencial de emergência sem controle de rotação
  • Mesma senha de admin em múltiplos sistemas
OBRIGATÓRIO · PRÁTICAS CORRETAS
O que deve ser implementado
  • CyberArk, BeyondTrust, Delinea, HashiCorp Vault ou Azure PIM
  • Check-out de credencial com janela de tempo definida
  • Rotação automática após cada check-out
  • Log de quem acessou, quando e por quanto tempo
  • Alerta para qualquer acesso fora do horário padrão
  • Acesso ao vault protegido por MFA de fator forte
Processo de Check-out
Fluxo de Acesso via Vault

Todo acesso a conta privilegiada Tier 1 e Tier 2 deve seguir este fluxo.

EtapaAçãoEvidência Gerada
1 · Solicitação Operador autentica no vault com MFA e solicita acesso à conta Log de autenticação no vault com identidade do operador
2 · Justificativa Preenchimento obrigatório: motivo, sistema alvo, janela de tempo Registro de justificativa vinculado ao check-out
3 · Aprovação (Tier 1) Aprovação em tempo real por outro admin ou gestor — se exigido Registro do aprovador e timestamp
4 · Acesso Vault libera a sessão — operador nunca vê a senha em texto claro Sessão gravada (para Tier 1) / Log de acesso (Tier 2)
5 · Encerramento Sessão encerrada manualmente ou por expiração do tempo Log de encerramento com duração da sessão
6 · Rotação Vault rotaciona a credencial automaticamente após check-in Log de rotação com timestamp
