Controle 8.3 — Information Access Restriction
Matriz de Classificação da Informação

A restrição de acesso começa pela classificação. Sem saber o nível de sensibilidade do dado, não é possível definir quem pode acessá-lo.

NívelDefiniçãoExemplosQuem acessaControles obrigatórios
PÚBLICO Informação sem restrição de divulgação Site institucional, comunicados oficiais Qualquer pessoa Nenhum adicional
INTERNO Uso interno — não deve ser divulgado externamente Procedimentos, políticas, organigramas Colaboradores ativos Autenticação corporativa
CONFIDENCIAL Acesso restrito a funções específicas com need-to-know Dados de RH, financeiro, clientes, contratos Perfil com justificativa aprovada MFA + RBAC + log de acesso
RESTRITO Máxima restrição — dano severo se exposto Dados pessoais sensíveis, segredos de negócio, chaves Mínimo necessário — aprovação dupla MFA forte + masking + log imutável + revisão mensal
Information Owner: cada ativo de informação deve ter um responsável definido — a pessoa que aprova acessos, valida revisões periódicas e decide sobre a classificação. Sem Information Owner, o controle 8.3 não pode ser operacionalizado de forma auditável.
Proteção na Camada de Dados
Data Masking — o que aplicar e onde

Mascaramento de dados protege informações sensíveis mesmo quando o usuário tem acesso legítimo ao sistema — exibe apenas o necessário para a função.

EXPOSIÇÃO INDEVIDA
O que não deve acontecer
  • CPF completo visível para atendentes sem necessidade
  • Número de conta bancária exposto em relatórios gerais
  • Dado de saúde acessível a qualquer colaborador de RH
  • Interface mascarada, mas dado aberto no banco — sem proteção real
  • Log de acesso a dados sensíveis sem rastreabilidade por usuário
IMPLEMENTAÇÃO CORRETA
O que deve ser aplicado
  • CPF exibido como ***.456.789-** para perfis sem necessidade do dado completo
  • Número de conta mascarado na interface e na camada de dados
  • Dado de saúde acessível apenas ao perfil médico / jurídico com log
  • Mascaramento aplicado na camada de aplicação e banco — não só na UI
  • Acesso ao dado completo requer aprovação adicional e gera alerta

Segregação Lógica
Acesso à rede não implica acesso aos dados

A segregação lógica garante que estar autenticado no ambiente corporativo não equivale a ter visibilidade de todos os recursos. Cada camada tem sua própria barreira de acesso.

CamadaControle aplicadoQuem defineEvidência esperada
RedeVLAN por criticidade, firewall com regras por segmentoTI / InfraDiagrama de rede + regras de firewall
Sistema / AplicaçãoMenu e funcionalidades exibidos por perfil RBACDono da aplicaçãoConfiguração de roles na aplicação
Banco de DadosPermissões por schema, tabela ou coluna por usuário/serviçoDBA + Information OwnerGrant/Revoke de permissões documentado
Arquivo / DocumentoACL por pasta, SharePoint/Drive com controle por grupoGestor da áreaConfiguração de permissões + revisão periódica
APIScopes OAuth limitados à operação necessáriaEquipe de desenvolvimentoConfiguração de scopes + documentação de API
Ponto de atenção: Sysadmin com acesso irrestrito ao banco de dados de RH, Financeiro ou dados de saúde — mesmo com justificativa técnica — viola o princípio de need-to-know do controle 8.3. O acesso técnico necessário para manutenção deve ser separado do acesso aos dados em si, com sessão gravada e log imutável quando não for possível segregar completamente.
